Technical Specification · v2.0 · Content-First Pivot

Hearth
System Architecture v2

The content-first rebuild. Hearth is now a recipe discovery app with a smart pantry underneath — an image-based reels feed of Hearth’s own recipes, an Explore page, an AI cook-assistant, a pantry & shopping list, and a profile with collections. Three recipe origins with a hard public/private permission boundary. This document supersedes v1.

React Native / Expo 5-tab nav + center AI button Image reels feed + likes Supabase + RevenueCat Cloudflare R2 (images) expo-notifications (local) 3 recipe origins No video hosting
00 · What Changed & Why

From kitchen utility to
recipe app with a smart pantry.

v1 Hearth was pantry-first: scan receipt → Panic Button → cook. v2 Hearth is content-first: a feed of beautiful recipes is the front door, and the pantry/AI/waste features make cooking them effortless. The core food-waste engine survives underneath — it’s just no longer the first thing users see.

The two principles that still govern everything
1. The user never manually logs consumptionReceipt OCR sets quantities at purchase. Cooking a recipe auto-deducts via a Postgres trigger. The pantry maintains itself. This survives the pivot unchanged and still rules the data model.
2. Recipe origin is a permission boundary, not a labelNew in v2. Every recipe has an origin: hearth_featured, community, or user_import. The first two can be public; user_import is private forever, barred at the database (RLS) level. This is what keeps link-importing legal as the app scales.
Stack changes from v1 → v2
Dropped: OneSignalReplaced by expo-notifications (on-device local scheduling). Reminders fire on any day/time the user picks, scheduled locally. One fewer service, works offline.
Dropped: Branch.io + UpstashCreator affiliate links deferred (didn’t fit content-first model). Upstash cron replaced by Supabase pg_cron where any server scheduling is needed.
Added: Cloudflare R2 as image CDNThe feed is images, not video — cheap to host, fully controlled, no per-minute streaming cost. No Mux, no video infrastructure.
Whisper: demoted to optionalOnly used inside the private link-import flow to transcribe a video with no written recipe. Not in the core path. The main feed is text+image recipes with no transcription needed.
Kept: SupabaseDB, Auth, Storage, Realtime, Edge Functions, pg_cron. The spine. Non-negotiable.
Kept: RevenueCatNative IAP for subscriptions. Removing it means hand-rolling StoreKit + Play Billing — more work, not less. Stays.
01 · The Five Tabs

Navigation: 4 tabs + a center AI button.

The bottom bar has four standard tabs plus a prominent center button (Creme-style) that opens the AI cook-assistant as a full screen. Home is the front door — an image reels feed of Hearth recipes.

TAB 1
Home
Vertical reels-style scroll through image recipe cards (Hearth-featured now, +community later). Like button on each. Tap a card → Recipe Detail with ingredients, steps, time, nutrition, pantry-match, and “cook this”.
TAB 2
Explore
Recommended-for-you, theme filters (Asian, Italian…), search bar, and a “Cook from your pantry” row that matches Hearth recipes to what the user has. Pantry matching is deterministic — no AI needed.
CENTER
AI ✨
Prominent center button. Opens the AI cook-assistant. v1 ships the one-shot “I’m exhausted” button (pantry in → one recipe out). v2 adds conversational chat later. Allergens are hard constraints.
TAB 4
Pantry & List
Two sub-tabs. Pantry: items from onboarding, OCR, manual add. Shopping List: AI suggests low-stock items at the top as a confirm-to-add row — user must approve, nothing auto-added silently.
TAB 5
Profile
Collections (saved recipes + private imports, two sub-tabs), plus custom recipe upload. Settings: name, avatar, household sharing, subscription, allergies, stats, flexible reminder day/time.
Key navigation decisions
The old Panic Button movedIn v1 it lived on the Home screen. It is now the center AI tab’s one-shot mode. The logic you already built transfers almost directly — it just lives behind the center button now.
Saved recipes moved to ProfileThe old Explore/Saved tab is split: discovery (recommended/themes/search) is the new Explore; saved + imported recipes live in Profile → Collections.
Cook-from-pantry needs no AIIt’s the existing pantry_match_pct logic pointed at Hearth’s recipe library instead of imports. Deterministic, cheap, instant.
Center button is a modal routeNot a flat tab — it opens the AI assistant as a presented full-screen route so it feels like an action, matching the Creme reference.
02 · The Recipe Origin Model

Three origins. Two permission classes.
One legal boundary.

This is the most important new concept in v2. Every recipe has an origin that determines who can see it and whether it can ever go public. The boundary is enforced in Row Level Security — not just hidden in the UI — because it is what keeps link-importing legally defensible as Hearth scales.

Created by
Visible to
Can go public?
hearth_featured
Hearth / hired cooks
Everyone (home feed)
Yes — it is the feed
community
A user, original work
Self now; everyone Phase 2
Yes — their own work
user_import
Extracted from a link
Only the importer, forever
No — hard-barred by RLS
Why this boundary exists (the legal logic)
Recipes (facts) aren’t copyrightableA list of ingredients and functional steps is not protected. Extracting those facts from a video into a text card is the defensible core of what Flavorish-style import does.
Expression is protectedThe original video, photography, and exact prose are protected. So Hearth extracts facts (ingredients, steps, nutrition) but never rehosts the source video or copies its wording verbatim.
Private + user-initiated keeps it personalOne user importing their own link for their own private collection is transformative and personal. Promoting that to a public feed would turn it into republishing — which is why user_import can never go public.
Platform ToS is a separate riskPulling media off TikTok/IG may violate their terms regardless of copyright. So link-import is a secondary convenience, never a load-bearing pillar, and stays private and small.
Hard rule enforced in RLS & AGENTS.md
A recipe with origin = 'user_import' can never appear on the public feed, in Explore, in another user’s view, or be promoted to community. The RLS SELECT policy on the feed query structurally excludes it. This is not a UI toggle — it is a database-level guarantee. Any code path that would surface an imported recipe publicly is a critical defect. This single rule is what keeps the import feature defensible. Not legal advice — consult an IP lawyer in-market before monetizing at scale.

The cook trigger still rules — now across origins
cook_logs INSERT → Postgres trigger → deduct pantry quantities, mark depleted items used, increment items_rescued, recompute pantry_match_pct on affected collections. New in v2: because a cooked recipe may be hearth_featured, community, or user_import, its ingredients may include unmapped items (ingredient_id null, raw_text set). The trigger deducts only mapped ingredients and silently skips unmapped ones — never errors. Featured/community recipes are fully mapped at publish time; only messy imports carry unmapped rows.
05 · Edge Functions

Server-side logic. All AI lives here.

Plain reads/writes go through PostgREST with RLS. Anything touching an AI service, a secret, or multi-step logic is an Edge Function. Keys never reach the client.

ai-exhausted
Center AI button → “I’m exhausted”
The v1 one-shot. Reads pantry, returns one cookable recipe — first tries to match an existing Hearth recipe the user can make, falls back to generating one. Never uses ingredients not in pantry; allergens are hard constraints.
01Fetch active pantry_items + dietary_profile
02Detect state: empty / sparse / expiring / general
03First try: query published Hearth recipes with high pantry match (deterministic, no AI cost)
04If a strong match exists → return that recipe (cheapest path, promotes our content)
05Else: GPT-4o-mini generates a recipe from pantry only, cuisine-anchored
06Validate every ingredient against pantry; retry once stricter if it hallucinates
07Validate against allergens — reject + regenerate on any violation
08Return recipe; if generated, save as origin=ai_generated (private to user)
import-recipe
Profile → Collections → paste link (private only)
The Flavorish-style import. Extracts ingredients, steps, and nutrition from a pasted YT/IG/TikTok link into a private recipe card. Result is always origin=user_import — barred from public surfaces. Whisper only if no written recipe exists.
01Create import_job (status=queued), return immediately so UI can show progress
02Caption/description layer: GPT-4o-mini tries to extract from available text first
03If incomplete and audio is available: Whisper transcribes → GPT structures it
04Extract ingredients + steps + nutrition estimate into structured JSON
05Map ingredients to canonical IDs; keep raw_text for any that don’t map
06INSERT recipe with origin=user_import, created_by=user, source_url set
07Add to collections (type=imported); compute pantry_match_pct
08Update import_job (status=done); never rehost source media, store facts only
ocr-receipt
Pantry tab → scan receipt (survives from v1)
Unchanged from v1. Google Vision OCR → brand strip → canonical map → user confirms → batch insert to pantry. Ready-made products silently dropped.
01Base64 image → Google Vision DOCUMENT_TEXT_DETECTION
02Parse lines, strip brands, match to ingredients.aliases
03Unmatched / ready-made silently dropped
04Return parsed items for confirm screen → batch INSERT on confirm
publish-featured-recipe
Admin / creator tool — you adding launch recipes
How Hearth’s own recipes enter the feed. Takes your image + raw recipe text, AI structures it into a clean draft, you review, then publish. AI drafts — a human approves before it goes live.
01Upload hero image (+ optional step images) to R2
02GPT-4o-mini structures raw text into title, steps, ingredients, nutrition draft
03Map ingredients to canonical IDs
04INSERT recipe origin=hearth_featured, status=review
05You review/edit in an admin view → set status=published → appears in feed
suggest-shopping-items
Pantry → Shopping List sub-tab opens
Generates the “suggested low-stock” row at the top of the shopping list. Deterministic where possible. Inserts as is_confirmed=false — user must approve each before it joins the real list.
01Find pantry items below a low-stock threshold or frequently re-bought
02Insert as shopping_list_items source=ai_suggested, is_confirmed=false
03UI shows them in a distinct “suggested” row; confirm flips is_confirmed=true
cook-completion-trigger
Postgres trigger on cook_logs INSERT (v1)
Unchanged core. Deduct pantry, mark used, count rescued, recompute matches. Now skips unmapped ingredients gracefully for messy imports.
01For each mapped recipe ingredient: deduct quantity × servings
02Depleted → status=used; count Red Zone items rescued
03Recompute pantry_match_pct on affected collections; skip null-ingredient rows
06 · AI & Cost Discipline

Cheap by default. AI only where it earns it.

The content-first model actually reduces AI load: the feed is pre-made recipes (no per-view AI), pantry matching is deterministic, and the “I’m exhausted” button prefers retrieving an existing recipe over generating one.

Where AI fires — and where it deliberately doesn’t
Feed & Explore — zero AIRecipes are pre-made and stored. Browsing, theme filters, search, and the cook-from-pantry row are plain database queries. The most-used screens cost nothing in AI.
“I’m exhausted” — retrieve before generateFirst tries to match an existing Hearth recipe (free, promotes your content). Only generates with GPT-4o-mini when nothing fits. Most taps cost nothing.
Import — GPT-4o-mini, Whisper only if neededCaption text first (cheap). Whisper transcription only when there’s no written recipe. User-initiated and occasional, so volume is naturally low.
Conversational AI — Phase 2, Plus onlyThe multi-turn chat assistant (GPT-4o) is deferred and gated to Plus when it ships. v1 ships only the one-shot button.
Allergens — always validatedEvery AI output (exhausted button, import, generation) is cross-checked against the user’s allergens server-side. A leak is a critical defect, logged to Sentry, regenerated.
Free-tier limits“I’m exhausted” generation and link imports are metered on free tier (exact numbers in business plan). Browsing the feed is always unlimited — never paywall discovery.
07 · Key User Stories (v2)

The new core experiences.

The stories that define v2. Pantry, OCR, and cook-deduction stories from v1 still apply and aren’t repeated here — these are the new and changed ones.

US-V2-01
Scroll the recipe feed and like recipes
NewHome
“As a user opening Hearth, I want to scroll a beautiful feed of recipes like Instagram reels so I get inspired about what to cook.”
Acceptance Criteria
  • Home is a vertical, full-bleed, swipe-to-next feed of recipe image cards
  • Each card shows hero image, title, cuisine theme, cook time, and a like button with count
  • Feed shows only published recipes with origin hearth_featured (community added Phase 2)
  • user_import recipes never appear here — enforced by the RLS feed query, not UI filtering
  • Liking is optimistic, writes to likes table, increments like_count; double-tap and button both work
  • Tapping a card opens Recipe Detail
  • Feed uses FlashList for smooth scrolling; images lazy-load from R2
  • Empty feed (pre-launch, no recipes published): show a friendly “new recipes coming” state, never a blank screen
  • Image fails to load: show a placeholder, never a broken image icon
US-V2-02
View a recipe’s full breakdown with step view
ChangedRecipe Detail
“As a user who tapped a recipe, I want all the details — ingredients, time, nutrition, and a clear step-by-step view — so I can decide to cook it and follow along.”
Acceptance Criteria
  • Detail shows hero image, title, cook time, difficulty, servings, nutrition per serving
  • Ingredients list shows have/missing state against the user’s pantry (pantry match %)
  • “Add missing to shopping list” adds unmatched ingredients in purchase units
  • Step view (Creme-style): numbered steps, optional per-step image, optional inline timer
  • If the recipe has video_timestamps and a source video, steps can deep-link to that moment
  • Save to collection button (saved sub-tab); like button
  • “I cooked this” fires cook_log → trigger deducts pantry
  • Imported recipe with unmapped ingredients: show raw_text for those, still cookable, match % counts only mapped ones
US-V2-03
Import a recipe from a social link (private)
NewAI
“As a user who saw a recipe on TikTok, I want to paste the link and get a clean recipe card with ingredients, steps, and nutrition saved privately to my collection.”
Acceptance Criteria
  • Paste a YT/IG/TikTok link in Profile → Collections → Import
  • Progress shown immediately via import_job (queued → processing → done)
  • Result has ingredients, steps, and a nutrition estimate
  • Result is ALWAYS origin=user_import — private to the importer, never on any public surface
  • Saved to Collections (imported sub-tab); pantry match computed
  • Hearth extracts facts only — never rehosts the source video, never copies prose verbatim
  • Free tier import limit enforced; upgrade prompt at limit
  • No written recipe + has audio: Whisper transcribes; if still nothing usable, offer manual entry
  • Extraction fails: import_job=failed, user sees “couldn’t read this one” + manual entry option, never silent loss
  • Any attempt to make an imported recipe public is rejected at the database layer
US-V2-04
Upload my own original recipe
NewCommunity seed
“As a user proud of a dish I make, I want to add my own recipe with a photo, ingredients, and steps so I can keep it and (later) share it.”
Acceptance Criteria
  • Upload flow: photo + title + ingredients + steps + optional nutrition
  • Ingredients map to canonical IDs (with search), enabling pantry deduction when cooked
  • Saved as origin=community, status=draft/private to the user in Phase 1
  • In Phase 2, user can publish their community recipe to the public feed (it is their own work)
  • This is the clean, legal sharing path — distinct from user_import which can never be shared
  • User skips nutrition: it’s optional; can be AI-estimated later
US-V2-05
“I’m exhausted” — one tap, one meal
MovedAI
“As an exhausted user, I want to tap the center button and be told one thing to cook from what I have, so I don’t have to think.”
Acceptance Criteria
  • Center AI tab; “I’m exhausted” is the v1 one-shot action
  • Prefers matching an existing Hearth recipe the user can cook now (free, promotes content)
  • Falls back to GPT-4o-mini generation only when nothing fits
  • Uses only pantry ingredients; respects allergens absolutely
  • Returns one recipe with ingredients, steps, time; “I cooked this” available
  • Conversational chat mode is explicitly Phase 2 — do not build it in v1
  • Generation hallucinates a missing ingredient: validate, retry once, then honest sparse response
US-V2-06
Shopping list with confirm-to-add AI suggestions
ChangedPantry tab
“As a user maintaining my kitchen, I want the app to suggest low-stock items to rebuy, but I decide what actually goes on my list.”
Acceptance Criteria
  • Shopping List is the second sub-tab of the Pantry tab
  • Suggested low-stock items appear in a distinct row at the top, marked as suggestions
  • Suggestions are is_confirmed=false — nothing is silently added to the real list
  • User taps to confirm each suggestion; only then does it join the list
  • Items can be added from recipe “add missing”, manually, or from confirmed suggestions
  • Checking off a bought item; auto-remove when later scanned on a receipt
  • No low-stock items: suggestion row hidden entirely, not shown empty
US-V2-07
Flexible cooking reminder (any day/time)
ChangedLocal notif
“As a user, I want to choose which day and time Hearth reminds me to plan meals, not be locked to Sunday.”
Acceptance Criteria
  • Reminder day and time are user-configurable in Profile → Settings
  • Scheduled on-device via expo-notifications — no server push, no OneSignal
  • Reminder content highlights expiring items / suggests cooking from pantry
  • User can disable reminders entirely
  • reminder_prefs persisted to users table so it survives reinstall (re-scheduled on launch)
  • Notification permission denied: settings explain it and deep-link to OS settings
